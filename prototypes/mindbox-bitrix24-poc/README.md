# Bitrix24 → Mindbox integration PoC

Демонстрационный сервис для предметного обсуждения пилота интеграции Bitrix24 и Mindbox.

## Что уже реализовано

- приём событий `ONCRMCONTACTUPDATE` и `ONCRMDEALUPDATE` от Bitrix24;
- разбор стандартной bracket-нотации webhook-полей;
- получение актуальных данных контакта и сделки через входящий webhook Bitrix24;
- преобразование контакта и сделки в тело операции Mindbox API 3.0;
- отправка операции с авторизацией по secret key;
- повторные попытки при `429` и серверных ошибках;
- защита входящего endpoint общим секретом;
- краткоживущая дедупликация повторных событий;
- unit- и HTTP-тесты;
- Dockerfile и проверка переменных окружения.

## Сценарий пилота

1. Bitrix24 отправляет событие изменения контакта или сделки.
2. Сервис получает полную сущность через REST API Bitrix24.
3. Маппер собирает клиента Mindbox и дополнительные поля сделки.
4. Сервис вызывает настроенную операцию Mindbox.
5. Временные ошибки повторяются с экспоненциальной задержкой.

## Запуск

```bash
cp .env.example .env
npm install
npm run dev
```

Проверка:

```bash
npm run check
```

Webhook URL для Bitrix24:

```text
https://your-host.example/webhooks/bitrix?secret=<WEBHOOK_SHARED_SECRET>
```

## Что нужно согласовать перед коммерческим пилотом

- точные события и направление синхронизации;
- операция и endpoint Mindbox;
- сопоставление идентификаторов, полей клиента и сделки;
- правила создания/обновления клиента;
- обязательные пользовательские поля в конкретных аккаунтах;
- хранение очереди и дедупликации в Redis/PostgreSQL вместо памяти;
- требования к журналированию, персональным данным и размещению;
- критерии приёмки и тестовый контур.

## Ограничение PoC

Проект не содержит ключей и не обращается к реальным аккаунтам. Для полноценной проверки потребуются тестовые доступы и утверждённая схема операции Mindbox.
