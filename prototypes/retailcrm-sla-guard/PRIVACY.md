# Конфиденциальность и безопасность RetailCRM SLA Guard

## Какие данные обрабатываются

Для контроля SLA модуль получает через RetailCRM API только данные, необходимые для определения просрочки:

- внутренний идентификатор и номер заказа;
- текущий статус;
- время последнего изменения статуса;
- технические данные пагинации и ответа API.

Имя клиента и сумма заказа могут использоваться только для расширенного текста Telegram-уведомления. Эта настройка по умолчанию выключена и включается владельцем аккаунта явно.

## Куда передаются данные

- RetailCRM API используется для чтения заказов и управления экземпляром интеграционного модуля.
- Telegram Bot API используется для доставки уведомлений в чат, указанный владельцем аккаунта.
- Иные внешние получатели не используются.

При выключенной настройке расширенных данных в Telegram передаются только номер заказа, статус и длительность превышения SLA.

## Хранение

- RetailCRM API-ключи и Telegram-токены хранятся зашифрованными с использованием AES-256-GCM.
- Ключ шифрования хранится отдельно в секретах развёртывания и не включается в репозиторий.
- Для защиты от повторных уведомлений хранится ограниченный список технических ключей вида «заказ — статус — время изменения», не более 5 000 записей на аккаунт.
- Журналы содержат время, тип технического события, HTTP-статус, длительность запроса и сокращённый идентификатор клиента. API-ключи, Telegram-токены и содержимое заказов в журналы не записываются.
- Рекомендуемый срок хранения журналов — 30 дней. В репозитории приложена конфигурация `logrotate` для ежедневной ротации и удаления старых журналов.
- Конфигурация клиента хранится до отключения сервиса, расторжения договора или письменного запроса на удаление.

## Удаление данных

Владелец аккаунта может запросить удаление конфигурации и технического состояния по email `nikidom123@gmail.com` или в Telegram `@tahioff`. После проверки принадлежности аккаунта данные удаляются в разумный технический срок.

Деактивация или заморозка модуля немедленно останавливает обработку заказов, но сама по себе не удаляет конфигурацию, чтобы настройки можно было восстановить после повторной активации.

## Меры защиты

- проверка HMAC-SHA256 при первичном подключении;
- непредсказуемый `clientId` для идентификации аккаунта;
- проверка HTTPS и официального списка доменов RetailCRM;
- минимально необходимые разрешения API: `order_read` и `integration_write`;
- шифрование секретов при хранении;
- таймауты и ограниченные повторные попытки внешних запросов;
- отсутствие паролей пользователей RetailCRM;
- отсутствие секретов и клиентских данных в публичном репозитории;
- минимизация персональных данных в уведомлениях по умолчанию.

## Ответственность клиента

Клиент должен:

- создать отдельного Telegram-бота для рабочих уведомлений;
- ограничить доступ к Telegram-чату только уполномоченными сотрудниками;
- не публиковать токен бота;
- включать передачу имени клиента и суммы только при наличии законного основания и подходящих внутренних правил обработки персональных данных;
- своевременно сообщать о компрометации токена или изменении состава получателей.

# Privacy summary

The module reads order identifiers, status and status update time from RetailCRM. Customer name and order total are excluded from Telegram alerts by default and can only be enabled explicitly by the account owner.

RetailCRM API keys and Telegram bot tokens are encrypted at rest with AES-256-GCM. Logs do not contain credentials or order contents and should be rotated after 30 days. Data is not sold or shared with advertisers or unrelated third parties.
