# Чек-лист модерации RetailCRM SLA Guard

## 1. Что уже готово

- работа через RetailCRM API v5;
- простое подключение через `configUrl` и `registerUrl`;
- HMAC-SHA256 проверка `register[token]`;
- регистрация экземпляра через `POST /api/v5/integration-modules/{code}/edit`;
- обработка активации, деактивации, заморозки, разморозки и изменения URL аккаунта;
- личный кабинет настроек, открываемый POST-запросом с `clientId`;
- минимальные разрешения `order_read` и `integration_write`;
- проверка HTTPS и актуального списка доменов RetailCRM;
- шифрование API-ключей и Telegram-токенов AES-256-GCM;
- понятные пользовательские ошибки без stack trace;
- остановка обработки выключенных и замороженных аккаунтов;
- повторные попытки при 429, 5xx и временных сетевых ошибках;
- техническое логирование без секретов и содержимого заказов;
- 30-дневная схема ротации журналов;
- минимизация персональных данных в Telegram по умолчанию;
- автоматические unit- и smoke-тесты в GitHub Actions;
- русские и английские материалы карточки;
- политика поддержки и конфиденциальности;
- отдельный SVG-логотип без элементов фирменного стиля RetailCRM: `assets/logo.svg`;
- публичная интерактивная демонстрация: `https://terratectra.github.io/autotools-hub/site/`.

## 3. Методы API для комментария модератору

Модуль использует:

- `GET /api/v5/orders` — чтение заказов и времени изменения статуса;
- `POST /api/v5/integration-modules/{code}/edit` — регистрация и активация экземпляра модуля.

Запрашиваемые разрешения:

- `order_read`;
- `integration_write`.

Модуль не изменяет заказы, клиентов, статусы, задачи или пользователей.

## 4. Возможности для комментария модератору

1. Контроль времени нахождения заказа в каждом настроенном статусе.
2. Индивидуальный SLA в минутах для разных статусов.
3. Telegram-уведомления при превышении SLA.
4. Защита от повторного уведомления по тому же состоянию заказа.
5. Автоматическая приостановка при деактивации или заморозке платного модуля.
6. Поддержка изменения адреса RetailCRM-аккаунта.
7. Безопасное простое подключение без ручной передачи API-ключа владельцем аккаунта.
8. Шифрование секретов при хранении.
9. Уведомления без имени клиента и суммы заказа по умолчанию.
10. Техническое логирование для диагностики интеграции.

## 6. Сценарий ручной проверки

1. Открыть карточку неопубликованного модуля в партнёрском CRM-аккаунте.
2. Нажать «Подключить» и убедиться, что выполняется автоматическая регистрация.
3. На странице настроек указать тестовый Telegram chat ID, токен бота и правило `new:1`.
4. Создать или выбрать тестовый заказ в статусе `new` с временем изменения старше одной минуты.
5. Дождаться уведомления в Telegram.
6. Убедиться, что имя клиента и сумма не включены по умолчанию.
7. Повторно дождаться цикла и убедиться, что уведомление не дублируется.
8. Изменить статус заказа и вернуть его в контролируемое состояние — новое состояние должно иметь отдельный ключ уведомления.
9. Деактивировать модуль — опрос должен прекратиться.
10. Активировать модуль — опрос должен возобновиться.
11. Проверить callback заморозки и разморозки.
12. Проверить понятную ошибку при неверном Telegram-токене или недостатке средств для платной активации.
