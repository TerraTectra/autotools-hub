# Devnet Key Management

Devnet keys are test credentials, but they are still signing credentials. Treat them as secrets because leaked keys can corrupt shared test state, invalidate demos, consume provider quotas, and confuse incident analysis.

## What may be public

- wallet public address;
- cluster name;
- transaction signature;
- program ID;
- confirmed balance;
- non-secret diagnostic timestamps and error classes.

## What must remain secret

- keypair JSON arrays;
- seed phrases and mnemonic words;
- base58/private-key exports;
- JWTs and API tokens;
- RPC URLs containing provider credentials;
- signed authentication challenges that can be replayed;
- temporary artifact download URLs.

## Preferred storage order

1. Organisation secret manager or cloud KMS.
2. Repository/environment secret with access controls.
3. Protected CI artifact with short retention for an autonomous test wallet.
4. Local file outside the repository with restrictive permissions.

A protected artifact is acceptable for a disposable devnet identity when a stronger secret store is unavailable, but it needs explicit retention, restore ordering, and repository access controls.

## Restore-before-create pattern

Incorrect:

```text
create wallet → try to restore artifact → overwrite one of them
```

Correct:

```text
restore protected state → validate keypair → create only if absent → record public address
```

After restoration, verify:

- JSON is an array of valid byte values;
- the keypair can be reconstructed;
- derived public address matches the recorded address;
- the wallet is on the intended cluster;
- the artifact is from the expected repository/workflow lineage.

## File permissions

On Unix-like runners:

```bash
umask 077
mkdir -p private/devnet
chmod 700 private/devnet
chmod 600 private/devnet/wallet.json
```

On Windows, place the file in a user-private directory and restrict ACLs rather than relying on Unix mode bits.

## Git protection

Add broad secret patterns to `.gitignore`:

```gitignore
private/
*.keypair.json
*.wallet.json
.env
.env.*
```

Do not rely only on `.gitignore`. CI should also fail when staged changes contain known secret paths or obvious 64-byte key arrays.

Before publishing a patch:

```bash
git diff --cached --name-only
git diff --cached --check
```

Use a secret scanner when available.

## Logging rules

Log:

```text
Devnet wallet: Fz67...8FrD
Balance: 20000000 lamports
```

Never log:

```text
Wallet secret: [34, 91, ...]
RPC URL: https://provider.example/?api-key=SECRET
Authorization: Bearer eyJ...
```

Redact URL credentials before printing:

- remove username/password;
- remove query parameters named `key`, `api-key`, `token`, `auth`, or similar;
- print only protocol and hostname by default.

## Artifact design

Separate artifacts when possible:

- `devnet-wallet-state`: keypair and service credentials, protected and short-lived;
- `devnet-diagnostics`: public address, balance, classification, logs with redaction;
- `application-snapshot`: sanitised data consumed by a demo.

If one artifact must contain all files, never publish its download URL or attach it to a public release.

## Rotation

Rotate a devnet wallet when:

- secret exposure is suspected;
- an untrusted workflow could access it;
- test state is irreparably polluted;
- the external service binds identity incorrectly;
- the project intentionally starts a new isolated environment.

Do not rotate merely because a faucet request failed. A new wallet rarely bypasses an IP/account rate limit and destroys continuity.

## Human handoff

When a human must fund or verify a wallet, provide only:

- public address;
- cluster (`devnet` or `testnet`);
- official action/page;
- minimum required amount;
- verification method.

Never ask the human to import a key generated by the agent into a browser wallet unless the workflow explicitly requires custody transfer and a safe out-of-band mechanism exists.
