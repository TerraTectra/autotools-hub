Процесс от входа до маршрутизации.

Логику можно вызывать из RPA-платформы, cron, контейнера или другого оркестратора. Бизнес-исключения не превращаются в техническое падение всего пакета.

01 · ВходДва CSV: заказы и платежи с обязательной проверкой колонок и значений.
02 · СверкаУспешные статусы платежей группируются по заказу, суммы считаются через Decimal.
03 · ИсключенияПроблемы фиксируются отдельными кодами и не прерывают обработку остальных записей.
04 · ВыходCSV результата, CSV очереди исключений и JSON-сводка для следующего шага.

Какие ситуации обнаруживает робот.

Каждая ветка воспроизводится синтетическими данными и покрыта unit-тестами.

MISSING_PAYMENT

У заказа нет ни одного успешного платежа.

AMOUNT_MISMATCH

Фактически оплаченная сумма не совпадает с суммой заказа.

DUPLICATE_PAYMENT

Для одного заказа найдено несколько успешных платежей.

ORPHAN_PAYMENT

Платёж ссылается на неизвестный заказ.

Артефакты, которые можно проверить.

Проект не требует доступа к рабочим системам и не содержит персональных данных.

Выход процесса

Файлы для людей и систем

  • reconciliation.csv — итог по каждому уникальному заказу;
  • exceptions.csv — очередь ручной проверки или следующего робота;
  • summary.json — метрики для оркестратора, мониторинга или уведомления.
Контроль качества

Воспроизводимый CI

  • Python 3.12 в чистом GitHub Actions worker;
  • полный набор unit-тестов;
  • запуск робота на синтетических CSV;
  • отдельная проверка значений итоговой JSON-сводки.

Честная граница демонстрации

Это platform-neutral Python/RPA-пруф, а не заявление о коммерческом опыте в PIX, Primo, Sherpa или другой конкретной студии роботов. Проект показывает проектирование процесса, обработку исключений, тестирование и готовность переносить логику в выбранную платформу.
